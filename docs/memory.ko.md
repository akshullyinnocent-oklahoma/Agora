# 메모리 & 캐시

Agora는 모델이 대화 간에 정보를 기억할 수 있도록 영구 메모리 시스템을 갖추고 있습니다. 자동 임베딩 기반 캐싱과 결합하여 사용량에 따라 성장하는 지식 베이스를 제공합니다.

## 메모리 유형

### 활성 메모리

모델에 대한 **모든 API 호출**에 포함되는 단일의 항상 켜져 있는 메모리 컨텍스트입니다. 모델이 항상 볼 수 있는 스티커 메모라고 생각하세요.

**활성 메모리 사용처:**
- 이름, 기본 설정, 배경 정보
- 모델이 항상 알아야 할 프로젝트 컨텍스트
- 모든 대화에 적용되는 상시 지침
- 반복해서 말하기 지루한 사실들

**활성 메모리 내용 예시:**
```text
사용자: Newo Ether
선호 사항: 일상 대화는 중국어, 기술 주제는 영어 선호.
프로젝트: Agora 개발 중 — BYOK Android LLM 클라이언트.
코딩 스타일: Kotlin, Jetpack Compose, MVVM 아키텍처.
```

#### 활성 메모리 편집

1. **설정 → 메모리**로 이동
2. **활성 메모리**로 스크롤
3. **활성 메모리 편집** 탭
4. 내용 입력
5. **저장** 탭

**활성 메모리 접근**이 활성화된 경우 모델도 도구 호출을 통해 활성 메모리를 업데이트할 수 있습니다.

---

### 저장된 메모리

모델이 검색, 읽기, 생성, 편집, 삭제할 수 있는 이름 지정된 메모리 파일 모음입니다. 항상 전송되는 활성 메모리와 달리, 저장된 메모리는 필요에 따라 검색됩니다.

**저장된 메모리 사용처:**
- 참조 자료 (API 문서, 설정 세부 정보, 명령어)
- 프로젝트별 노트
- 과거 대화에서 얻은 교훈과 통찰
- 관련성이 있을 때 모델이 기억했으면 하는 모든 것

#### 수동으로 메모리 생성

1. **설정 → 메모리**로 이동
2. **메모리 추가** 탭
3. 입력:
    - **제목** — 설명적인 이름
    - **설명** — 간략한 요약 (검색 매칭에 사용)
    - **내용** — 전체 메모리 내용
4. **생성** 탭

#### 모델 생성 메모리

**저장된 메모리 접근**이 활성화된 경우, 모델은 도구 호출을 통해 메모리 파일을 생성, 읽기, 업데이트, 삭제할 수 있습니다. 이를 통해 모델은:

- 사용자가 알려준 사실을 기억
- 유용한 코드 스니펫이나 설정 저장
- 시간이 지남에 따라 지식 베이스 구축
- 오래된 정보 정리

---

## 메모리 권한

모델이 접근할 수 있는 것을 제어합니다:

| 설정 | 위치 | 활성화 시기 |
|---------|----------|----------------|
| **저장된 메모리 접근** | 설정 → 메모리 | 모델이 메모리 파일을 읽고/쓰도록 하려는 경우 |
| **활성 메모리 접근** | 설정 → 메모리 | 모델이 영구 컨텍스트를 업데이트하도록 하려는 경우 |
| **과거 대화 접근** | 설정 → 대화 검색 | 모델이 채팅 기록을 검색하도록 하려는 경우 |

세 가지 모두 기본적으로 **꺼짐**입니다. 필요한 것만 활성화하세요.

## 자동 캐시

자동 캐싱은 새 메시지가 도착할 때 자동으로 임베딩을 생성합니다. 이를 통해 수동 개입 없이 대화 검색 인덱스를 최신 상태로 유지합니다.

### 자동 캐시 활성화

1. **설정 → 대화 검색**으로 이동
2. 임베딩 모델 선택 (아직 선택하지 않은 경우 — [임베딩 / RAG](embedding.ko.md) 참조)
3. **캐싱** 아래에서 **새 메시지 자동 캐시** 토글

활성화 시, 모든 새 메시지(사용자 및 모델)가 자동으로 임베딩되어 시맨틱 검색용으로 인덱싱됩니다.

### 수동 캐싱

자동 캐시가 꺼져 있는 경우 수동으로 메시지를 캐시할 수 있습니다:

1. **설정 → 대화 검색**으로 이동
2. **캐시** 탭 — 모든 캐시되지 않은 메시지의 임베딩을 계산
3. 진행 상황이 원형 표시기로 표시됨

**재캐시**를 탭하여 전체 인덱스를 처음부터 다시 빌드합니다. 모든 캐시된 임베딩을 삭제하고 모든 메시지를 다시 처리합니다. 다음 경우에 사용:
- 임베딩 모델을 변경한 경우
- 캐시가 손상되었거나 오래된 것 같은 경우
- 검색 결과가 예상치 못하게 나쁜 경우

!!! warning
    재캐싱은 되돌릴 수 없으며 메시지 수와 임베딩 모델 속도에 따라 시간이 걸릴 수 있습니다.

### 캐시 상태

임베딩 모델 설정은 캐시된 메시지와 캐시되지 않은 메시지 수를 보여줍니다:
- **"N개 메시지 모두 캐시됨"** — 최신 상태
- **"Y개 중 X개 메시지 캐시 안 됨"** — 처리 대기 중

## 채팅에서의 메모리 도구 호출

모델이 메모리 도구를 사용하면 인라인 카드가 표시됩니다:

| 도구 | 카드 텍스트 |
|------|-----------|
| 조회 | "N개의 저장된 메모리를 검색함" |
| 읽기 | "[메모리 이름] 읽음" |
| 저장 | "[메모리 이름] 저장됨" |
| 편집 | "[메모리 이름] 업데이트됨" |
| 삭제 | "[메모리 이름] 제거됨" |
| 활성 업데이트 | "활성 메모리 업데이트됨" |

카드를 탭하여 읽거나 쓴 전체 내용을 확인하세요.

---

## 모범 사례

- **활성 메모리를 간결하게 유지** — 모든 API 호출에 포함되므로 장황한 내용은 토큰을 낭비합니다
- **저장된 메모리에 설명적인 제목 사용** — 제목은 모델이 올바른 메모리를 찾는 데 도움이 됩니다
- **대화 검색을 정기적으로 사용하는 경우 자동 캐시 활성화**
- **임베딩 모델 전환 후 재캐시** — 다른 모델은 호환되지 않는 임베딩을 생성합니다
