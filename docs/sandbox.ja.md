# サンドボックス

Agora はデバイス上でローカルに軽量の Alpine Linux 環境を実行できます — インターネット接続は不要です。サンドボックスにより、モデルは隔離されたルートファイルシステム内でパッケージのインストールやコマンドの実行を行えます。

!!! note "利用条件"
    サンドボックスはシェル対応ビルドで利用可能です。**設定 → シェル → サンドボックス管理** からアクセスします。

## 仕組み

サンドボックスは、デバイスのアプリ専用ストレージにデプロイされた Alpine Linux ルートファイルシステムを使用します。最小限の `apk` ベースのパッケージマネージャーでこの環境にソフトウェアをインストールでき、コマンドは proot ベースのコンテナ内で実行されます。

これは完全な仮想マシンでは**ありません** — ホストカーネルを共有する軽量のユーザー空間コンテナです。リソース使用量を低く抑えながら、安全な実験に十分な隔離を提供します。

---

## VPN の干渉 — 重要

!!! danger "サンドボックスネットワーキングを使用する前に VPN をオフにしてください"

    VPN アプリケーションは proot の DNS 解決に干渉します。その理由は次のとおりです：

    **根本原因 — PRoot にはネットワーク名前空間の隔離がありません。**

    PRoot は `ptrace` を使用してシステムコールをインターセプトし、ファイルパスをリダイレクトしますが、`CLONE_NEWNET`（Linux ネットワーク名前空間）をサポートして**いません**。サンドボックス内のすべてのプロセスは、ホスト Android システムのネットワークスタックを直接共有します。仮想ネットワークインターフェース、隔離されたルーティングテーブル、独立した DNS 設定はありません。

    **Android の VPN が proot 内の DNS を破壊する仕組み：**

    1. Android VPN アプリは `VpnService` API を使用し、**TUN インターフェース**（proot 内部からのトラフィックを含む**すべての**デバイストラフィックをインターセプトする仮想ネットワークデバイス）を作成します
    2. 暗号化トンネル外への DNS 漏洩を防ぐために、VPN は**すべてのポート 53（DNS）トラフィック**を自身の DNS サーバーにリダイレクトします
    3. proot 内でアプリケーションが `getaddrinfo()`（標準 libc DNS リゾルバ）を呼び出すと、リクエストは Android のシステムリゾルバを経由します — これは VPN によって既にインターセプトされています
    4. Android 12+ では、Google が DNS リゾルバを作り直し、proot 環境内での `getaddrinfo()` が特に脆弱になりました（[termux/proot#215](https://github.com/termux/proot/issues/215)）
    5. VPN の TUN ルーティングとシステムリゾルバの DNS パスが proot 内で競合します：リゾルバが DNS クエリを送信し、VPN TUN がそれをインターセプトしますが、応答は proot の `ptrace` レイヤーを通じて戻ってきません

    **観測される症状：**

    | 操作 | 結果 |
    |-----------|--------|
    | `ping 1.1.1.1` | ✅ 動作（直接 IP、DNS 不要） |
    | `ping google.com` | ❌ 失敗 — 「名前解決の一時的な失敗」 |
    | `apk add python3` | ❌ 失敗 — `dl-cdn.alpinelinux.org` を解決できません |
    | `curl https://example.com` | ❌ 失敗 — 名前解決エラー |
    | `curl https://1.1.1.1` | ✅ 動作（IP 直接接続） |

    **修正方法:** サンドボックスでネットワーク操作（パッケージのインストール、`curl`、`wget` など）を行う前に、VPN を完全にオフにしてください。ネットワーク操作の完了後に VPN を再有効化できます。

    これは proot アーキテクチャの根本的な制限です — Android VPN が TUN インターフェースを介してシステムの DNS ルーティングを上書きする場合、ネットワークスタックを仮想化できません。

## セットアップ

### ルートファイルシステムのインストール

サンドボックスを初めて開くと、rootfs がインストールされていないことを示すダッシュボードが表示されます。**インストール** をタップして Alpine ルートファイルシステムをダウンロードして展開します。

!!! info "ストレージ使用量"
    ベースの rootfs は約 100～200 MB を使用します。インストールされたパッケージは追加の容量を消費します。合計ディスク使用量はダッシュボードに表示されます。

---

## パッケージ管理

### パッケージのインストール

1. テキストフィールドにパッケージ名を入力します（例：`python3`）
2. **インストール** をタップします
3. インストールの進行状況をターミナル出力で確認します

または、よく使うパッケージの **クイックインストールチップ** をタップします：

```
python3   git      curl      wget
openssh   nodejs   build-base   htop
```

### インストール済みパッケージ

インストールセクションの下に、すべてのインストール済みパッケージが以下とともに表示されます：

- **名前** — Alpine パッケージ名
- **バージョン** — インストールされたバージョン
- **説明** — 簡単な概要（省略表示）

### パッケージの削除

インストール済みパッケージの :material-close: アイコンをタップして削除します。削除前に確認ダイアログが表示されます。

---

## ダッシュボード

サンドボックスの準備ができると、ダッシュボードに以下が表示されます：

- **ディスク使用量** — プログレスバーと数値表示（MB または GB）
- **インストール数** — パッケージの総数

---

## ターミナル出力

パッケージのインストールまたは削除時に、ターミナル出力が入力フィールドの下のダークテーマのスクロール可能な等幅ビューに表示されます。出力は最新行を追跡するように自動スクロールします。

これを使用して：
- インストールの進行状況を監視
- 失敗したパッケージ操作をデバッグ
- パッケージがインストールするファイルを確認

---

## サンドボックスのリセット

下部の **デンジャーゾーン** には **サンドボックスをリセット** オプションがあります。これによりルートファイルシステムとインストールされたすべてのパッケージが完全に削除されます。

!!! danger "破壊的操作"
    サンドボックスをリセットすると、Alpine 環境全体が削除されます。その後、rootfs とすべてのパッケージを再インストールする必要があります。確認ダイアログが誤ったリセットを防ぎます。
