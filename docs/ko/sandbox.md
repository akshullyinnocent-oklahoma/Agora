# 샌드박스

Agora는 기기에서 로컬로 경량 Alpine Linux 환경을 실행할 수 있습니다 — 인터넷 연결이 필요 없습니다. 샌드박스를 통해 모델이 격리된 루트 파일 시스템에서 패키지를 설치하고 명령을 실행할 수 있습니다.

!!! note "이용 가능 여부"
    샌드박스는 모든 빌드에서 사용 가능합니다. **설정 → 셸 → 샌드박스 관리**에서 접근하거나 직접 **설정 → 샌드박스**에서 접근하세요.

## 작동 방식

샌드박스는 기기의 앱 전용 저장소에 배포된 Alpine Linux 루트 파일 시스템을 사용합니다. 최소 `apk` 기반 패키지 관리자를 통해 이 환경에 소프트웨어를 설치할 수 있으며, 명령은 proot 기반 컨테이너 내에서 실행됩니다.

이는 완전한 가상 머신이 **아닙니다** — 호스트 커널을 공유하는 경량 사용자 공간 컨테이너입니다. 리소스 사용량을 낮게 유지하면서 안전한 실험을 위한 충분한 격리를 제공합니다.

---

## VPN 간섭 — 중요

!!! danger "샌드박스 네트워킹 사용 전에 VPN을 끄세요"

    VPN 애플리케이션이 proot의 DNS 해석을 방해합니다. 이유는 다음과 같습니다:

    **근본 원인 — PRoot에는 네트워크 네임스페이스 격리가 없습니다.**

    PRoot는 `ptrace`를 사용하여 시스템 콜을 가로채고 파일 경로를 리디렉션하지만, **`CLONE_NEWNET`**(Linux 네트워크 네임스페이스)을 지원하지 않습니다. 샌드박스 내의 모든 프로세스는 호스트 Android 시스템의 네트워크 스택을 직접 공유합니다. 가상 네트워크 인터페이스, 격리된 라우팅 테이블, 독립적인 DNS 구성이 없습니다.

    **Android의 VPN이 proot 내부에서 DNS를 차단하는 방식:**

    1. Android VPN 앱은 `VpnService` API를 사용하여 **TUN 인터페이스**를 생성합니다 — 이는 proot 내부의 트래픽을 포함한 **모든** 기기 트래픽을 가로채는 가상 네트워크 장치입니다
    2. 암호화된 터널 외부로의 DNS 누출을 방지하기 위해, VPN은 모든 포트 53(DNS) 트래픽을 자체 DNS 서버로 **리디렉션**합니다
    3. proot 내부에서 애플리케이션이 `getaddrinfo()`(표준 libc DNS 리졸버)를 호출하면, 요청은 VPN이 이미 가로챈 Android의 시스템 리졸버를 통과합니다
    4. Android 12+에서 Google은 DNS 리졸버를 재작업하여 proot 환경 내 `getaddrinfo()`를 특히 취약하게 만들었습니다 ([termux/proot#215](https://github.com/termux/proot/issues/215))
    5. VPN의 TUN 라우팅과 시스템 리졸버의 DNS 경로가 proot 내부에서 충돌합니다: 리졸버가 DNS 쿼리를 보내고, VPN TUN이 이를 가로채지만, 응답이 proot의 `ptrace` 계층을 통해 돌아오지 않습니다

    **관찰된 증상:**

    | 작업 | 결과 |
    |-----------|--------|
    | `ping 1.1.1.1` | ✅ 작동 (직접 IP, DNS 불필요) |
    | `ping google.com` | ❌ 실패 — "이름 해석 일시적 실패" |
    | `apk add python3` | ❌ 실패 — `dl-cdn.alpinelinux.org` 해석 불가 |
    | `curl https://example.com` | ❌ 실패 — 이름 해석 오류 |
    | `curl https://1.1.1.1` | ✅ 작동 (IP 직접 연결) |

    **해결책:** 샌드박스에서 네트워크 작업(패키지 설치, `curl`, `wget` 등)을 수행하기 전에 VPN을 완전히 끄세요. 네트워크 작업 완료 후 VPN을 다시 활성화할 수 있습니다.

    이는 proot 아키텍처의 근본적인 한계입니다 — Android VPN이 TUN 인터페이스를 통해 시스템의 DNS 라우팅을 재정의할 때 네트워크 스택을 가상화할 수 없습니다.

## 설정

### 루트 파일 시스템 설치

샌드박스를 처음 열면 rootfs가 설치되지 않았음을 나타내는 대시보드가 표시됩니다. **설치**를 탭하여 Alpine 루트 파일 시스템을 다운로드하고 추출합니다.

!!! info "저장소 사용량"
    기본 rootfs는 약 100–200 MB를 사용합니다. 설치된 패키지는 추가 공간을 소비합니다. 총 디스크 사용량은 대시보드에 표시됩니다.

---

## 패키지 관리

### 패키지 설치

1. 텍스트 필드에 패키지 이름 입력 (예: `python3`)
2. **설치** 탭
3. 터미널 출력에서 설치 진행 상황 확인

또는 일반적인 패키지의 **빠른 설치 칩**을 탭하세요:

```
python3   git      curl      wget
openssh   nodejs   build-base   htop
```

### 설치된 패키지

설치 섹션 아래에 모든 설치된 패키지가 다음 정보와 함께 나열됩니다:

- **이름** — Alpine 패키지 이름
- **버전** — 설치된 버전
- **설명** — 간략한 요약 (축약됨)

### 패키지 제거

설치된 패키지의 :material-close: 아이콘을 탭하여 제거합니다. 삭제 전 확인 대화 상자가 나타납니다.

---

## 대시보드

샌드박스가 준비되면 대시보드에 다음이 표시됩니다:

- **디스크 사용량** — 진행 막대 및 숫자 표시 (MB 또는 GB)
- **설치된 수** — 총 패키지 수

---

## 터미널 출력

패키지 설치 또는 제거 시, 터미널 출력이 입력 필드 아래의 어두운 테마, 스크롤 가능한 모노스페이스 보기에 나타납니다. 출력은 최신 줄을 따라 자동 스크롤됩니다.

다음 용도로 사용:
- 설치 진행 상황 모니터링
- 실패한 패키지 작업 디버그
- 패키지가 설치하는 파일 확인

---

## 샌드박스 재설정

하단의 **위험 구역**에는 **샌드박스 재설정** 옵션이 있습니다. 이는 루트 파일 시스템과 설치된 모든 패키지를 완전히 제거합니다.

!!! danger "파괴적 작업"
    샌드박스를 재설정하면 전체 Alpine 환경이 삭제됩니다. 이후 rootfs와 모든 패키지를 다시 설치해야 합니다. 확인 대화 상자가 실수로 인한 재설정을 방지합니다.
